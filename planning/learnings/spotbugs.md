# SpotBugs

**Issue covered in:** #11 (backend CI ‚Äî static analysis stage)

---

## ELI5

SpotBugs analyzes your compiled Java code looking for patterns that are likely bugs. Not style issues ‚Äî actual bugs: places where a null pointer dereference could happen, a resource that might not get closed, a thread safety violation, or a security vulnerability. It's like a second pair of eyes that reads your bytecode and flags suspicious patterns.

---

## How It Works

Unlike Checkstyle (which reads source code), SpotBugs reads **bytecode** ‚Äî the compiled `.class` files. This means it must run after `mvn compile`. It matches bytecode patterns against a library of known bug signatures.

The tradeoff: it can catch more sophisticated issues than source analysis (it sees what the compiler actually generated), but it can't be run before compilation.

---

## Bug Categories

SpotBugs organizes findings into categories:

| Category | Examples |
|---|---|
| **Correctness** | Null dereferences, infinite loops, bad `equals()` implementations |
| **Security** | SQL injection risks, HTTP response splitting |
| **Performance** | Unnecessary object creation, inefficient string operations |
| **Bad practice** | Ignoring return values, finalizer misuse |
| **Multithreaded correctness** | Unsynchronized access to shared state |

---

## Effort and Threshold

Two key settings control behavior:

**`effort`** ‚Äî how hard SpotBugs looks:
- `Min` ‚Äî fast, finds obvious issues only
- `Default` ‚Äî balanced, good for CI
- `Max` ‚Äî thorough, slower, more false positives

**`threshold`** ‚Äî minimum severity to report:
- `Low` ‚Äî reports everything including minor style-adjacent findings
- `Medium` ‚Äî reports medium and high severity
- `High` ‚Äî reports only high-severity findings (real bugs, security issues)

For CI, `effort=Default` and `threshold=High` is the right starting point. It catches real problems without drowning the build in noise.

---

## Maven Plugin

```xml
<plugin>
    <groupId>com.github.spotbugs</groupId>
    <artifactId>spotbugs-maven-plugin</artifactId>
    <version>4.8.6.4</version>
    <configuration>
        <effort>Default</effort>
        <threshold>High</threshold>
    </configuration>
</plugin>
```

Run in CI: `mvn spotbugs:check -B` (must run after `compile`)

## SpotBugs vs Checkstyle vs PMD

| | SpotBugs | Checkstyle | PMD |
|---|---|---|---|
| Analyzes | Bytecode | Source code | Source code |
| Finds | Real bugs, security issues | Style violations | Code quality (dead code, complexity) |
| Needs compile? | Yes | No | No |
| False positives | Low at High threshold | Low with custom config | Medium |
| Portfolio signal | High ‚Äî catches real bugs | Medium ‚Äî shows care about consistency | Medium |

**Why both Checkstyle and SpotBugs:** They cover different ground. Checkstyle enforces that the code is written consistently; SpotBugs enforces that it's written correctly. The issue required "lint + static analysis" ‚Äî that's two distinct concerns, not one.

## Common Gotchas

**Must compile first.** Running `mvn spotbugs:check` without prior compilation will fail or analyze stale bytecode. Always run `mvn compile spotbugs:check` or chain it after a compile step.

**Lombok and SpotBugs.** SpotBugs sees Lombok-generated bytecode (getters, builders, etc.). It can occasionally flag Lombok-generated patterns as suspicious. At `threshold=High`, most Lombok false positives are filtered out. If specific false positives appear, they can be suppressed with `@SuppressFBWarnings`.

**`spotbugs:check` vs `spotbugs:spotbugs`.** The `spotbugs` goal generates the report. The `check` goal generates the report AND fails the build if violations exceed the threshold. Always use `check` in CI.

- Configured in `backend/pom.xml` at `effort=Default`, `threshold=High`
- Run as part of the lint and static analysis stage: `mvn compile checkstyle:check spotbugs:check -B`
- `compile` runs first in the same command to ensure bytecode exists for SpotBugs
